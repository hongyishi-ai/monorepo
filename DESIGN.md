# 红医师统一入口平台设计契约

本文件是 hongyishi.cn 的长期品牌与 UI/UX 契约。它参考 Open Design 对 `DESIGN.md` 的定位：设计系统不是说明文档，而是所有页面、组件、原型和新项目接入时共同遵守的品牌输入。

参考来源：https://github.com/nexu-io/open-design/blob/main/docs%2Fi18n%2FREADME.zh-CN.md

## 1. 平台目标

红医师不是单个工具站，而是训练伤防治、热射病防治、战场救护、博客与播客的统一入口平台。

核心要求：

- 单一入口：`hongyishi.cn` 首屏必须能进入当前保留的所有项目。
- 可扩展：新增项目优先通过项目注册表和共享 UI token 接入，不复制一套孤立设计。
- 保留原意：每个原始项目的内容创意、流程细节、知识结构和交互巧思都要保留。
- 功能不删减：重构 UI 时默认迁移功能，不把复杂功能简化成静态展示。
- 可信医疗工具感：界面可以大胆，但信息层级必须清楚，风险提示、边界说明和操作反馈必须明确。

## 1.1 技术栈统一原则

红医师统一的是产品平台、品牌系统、部署入口和质量门禁，不强制所有项目迁移到同一个前端框架。

- Portal 保持 Next 静态导出，负责首页、内容、SEO、项目注册和统一入口。
- FMS 保持 Vite + React SPA，负责本地数据、评估流程、PWA、图表和复杂交互。
- 热射病和 TCCC 保持静态 HTML/Tailwind PWA，短期通过共享 Tailwind preset、移动底栏、内容治理 banner 和 Cloudflare 构建注入获得统一体验。
- 只有当静态项目持续新增复杂状态、跨页数据、表单校验、IndexedDB、权限/同步或大量组件复用时，才迁移为 Vite + React。
- 新项目先选择最小合适运行时：内容型资料优先静态，复杂工具优先 Vite React，内容入口和平台页优先 Portal。

## 2. 视觉论点

红医师采用“战地医疗海报 + 医疗指挥台”的视觉方向：巨型中文块字、暗红主信号、米白纸底、战地深蓝分区、心电线和强网格，用有力量的海报结构承载复杂医疗内容。

首屏记忆点：

- 一眼看到“红医师”三个巨型中文品牌字和统一入口，而不是某个孤立子项目。
- 项目入口像战地医疗任务卡或工具舱，不像营销网站的普通三卡片。
- 视觉张力来自块字、纸张颗粒、心电线、粗线、色块、编号、状态，而不是渐变背景和装饰光斑。
- 海报参考中的“战伤救护 / 训练伤防治 / 热射病防治 / 红医师”应作为同一品牌家族，而不是四套独立美术风格。

## 3. 品牌色板

共享 token 源位于 `packages/config/brand/tokens.json`，TypeScript 导出位于 `packages/ui/src/brand/index.ts`，CSS 变量位于 `packages/ui/src/styles/variables.css`。Tailwind 统一 preset 位于 `packages/config/tailwind/index.cjs`，所有 Tailwind 项目必须通过 `@hongyishi/config/tailwind` 引入，不再各自复制品牌色、字体和圆角。

- 主信号红：`#D93025`，用于品牌、危险、重点入口、关键提示。
- 战地暗红：`#A51F18`，用于海报主视觉、强警示背景和品牌封面。
- 行动橙：`#FF6B00`，用于热风险、警示、下一步动作。
- 技术蓝：`#007AFF`，用于工具状态、诊断、系统反馈。
- 战地深蓝：`#12313C`，用于下半区块、训练伤、工具状态和深色信息面。
- 临床浅蓝：`#78C7E7`，用于医疗帐篷、生命体征、冷却与处置提示。
- 结构黄：`#FFD700`，用于编号、辅助高亮、模块定位。
- 墨黑：`#000000`，用于主文字、粗边框、强结构。
- 纸白：`#FFFFFF`，用于主要阅读底色。
- 米白纸底：`#F4ECDC`，用于首页、海报式区块和品牌背景。
- 中性灰：`#737373`，用于次级说明。

禁止事项：

- 不使用大面积紫蓝渐变作为默认品牌表达。
- 不使用玻璃拟态作为主要信息容器。
- 不用低对比灰字承载医疗关键信息。

## 4. 字体与排版

- 中文正文：`Noto Sans SC` 或系统中文黑体。
- 英文和编号：`Roboto Mono` 或系统等宽字体，用于项目类型、状态、版本、编号。
- 展示字：`Bebas Neue` 或 `Impact`，仅用于首屏品牌、强标题和项目大字，不进入正文。

排版规则：

- 标题要短，能扫描。
- 医疗说明文字优先清楚，不追求文学化。
- 按钮、标签、卡片标题不得因为中英文混排发生溢出。
- 移动端不使用 viewport 宽度缩放字体。

## 5. 布局系统

桌面端：

- 最大内容宽度保持在 `1280px` 左右。
- 首页结构为：品牌首屏、项目入口、平台说明、内容入口、底部。
- 每个项目入口应展示状态：已整合、外部运行、计划整合。
- 不使用卡片套卡片。页面分区用完整区块，卡片只用于单个项目或单个工具。

移动端：

- 375px 宽度是最低原型校验口径。
- 首页优先展示品牌、核心入口、状态，不堆长段说明。
- 项目入口采用单列或横向可扫结构，触控目标不小于 44px。
- 复杂工具保留流程，但拆成明确步骤，不能把多步医疗流程挤在一个屏幕。

## 6. 组件基线

项目卡片：

- 必须包含项目名、类型、描述、状态和入口。
- 内部项目不新开标签页，外部项目新开标签页并使用 `noopener noreferrer`。
- 图片可以保留原项目气质，但文字层必须可读。

按钮：

- 主按钮用于进入工具或开始流程。
- 次按钮用于查看文档、博客、播客或外部资料。
- 危险和医学警示动作必须有明确颜色和文案，不只依赖图标。

表单与流程：

- 医疗、训练、热风险输入必须有校验和错误反馈。
- 结果页必须说明适用范围、非诊断边界和下一步建议。
- 本地保存、离线、PWA 等原有能力要迁移，不能静默删除。

## 7. 动效与交互

动效要服务层级：

- 项目卡片可使用轻微位移、阴影压缩或图片缩放。
- 风险状态可使用颜色和结构变化，不使用持续闪烁。
- 移动端减少昂贵动效，避免大面积 `backdrop-filter`。

签名交互：

- “按下工具舱”的位移阴影用于入口卡片和关键按钮。
- 状态标签在 hover 或 focus 时显示更明确说明。
- 键盘焦点必须清晰可见。

## 8. 性能与安全基线

性能：

- 静态资源使用指纹文件和长缓存。
- Service Worker 和 manifest 使用短缓存或 `no-cache`。
- 大型子项目应逐步做代码分割，FMS 当前大 chunk 是后续优化项。
- 第三方 CDN 资源要逐步本地化或加完整性校验。

安全：

- 前端不得暴露生产密钥。
- 需要第三方 API 的项目优先通过 Cloudflare Pages Functions 代理。
- 静态站点必须生成 `_headers`，包含基础安全头和缓存策略。
- Pages Functions 必须在函数响应内加安全头，因为 `_headers` 不作用于函数响应。
- CSP 先以兼容原项目为底线，后续按页面收紧。

## 9. 新项目接入流程

新增项目时按以下顺序执行：

1. 在 `apps/<project-id>` 放置项目源码或静态资源。
2. 在 `apps/portal/src/lib/projects.ts` 增加项目记录。
3. 如果项目能静态部署，加入 `scripts/build-cloudflare.mjs` 的复制或构建流程。
4. 使用 `packages/config/brand/tokens.json`、`packages/ui/src/brand/index.ts` 和 `packages/ui/src/styles/variables.css` 的品牌 token，不单独定义另一套主色。
5. Tailwind 项目必须引入 `@hongyishi/config/tailwind`，只在本项目配置里追加必要的 content globs、插件和少量项目特定扩展。
6. 保留原项目核心内容、工作流、数据结构、离线能力和说明文档。
7. 补充构建测试、路径测试、安全头测试和至少一个关键用户流程测试。
8. 在 README 中登记访问路径、部署要求和必要环境变量。

当前优先级：

- 第一阶段：统一入口、Cloudflare 单站点部署、热射病与 FMS 内部路径整合。
- 第二阶段：统一首页和项目入口 UI，形成移动端原型。
- 第三阶段：逐步重构各子项目 UI，保留功能并迁移到共享品牌系统。
- 第四阶段：性能预算、安全审计、CSP 收紧、第三方依赖治理。
